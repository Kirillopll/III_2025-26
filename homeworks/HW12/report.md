# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

В работе использовался датасет временного ряда `S12-hw-dataset.csv`.  
Целью было спрогнозировать значение целевой переменной `target` на 1 шаг вперёд.

Данные были отсортированы по времени и разделены на `train / validation / test` строго хронологически, без перемешивания.

В рамках работы были сравнены четыре подхода:

- `B1` — наивный baseline `naive-last`
- `B2` — baseline `moving-average`
- `B3` — `Ridge` на лаговых, rolling- и календарных признаках
- `R1` — рекуррентная модель `GRU`

Лучшим подходом оказалась модель `GRU (R1)`, так как она показала наименьшее значение `MAE` на `validation` и затем была один раз проверена на `test`.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: регулярная временная последовательность
- Горизонт прогноза: `1` шаг вперёд
- Размер датасета: определяется автоматически в ноутбуке
- Диапазон дат: определяется автоматически в ноутбуке после преобразования колонки `date`
- Есть ли дополнительные внешние признаки: в основной постановке использовались производные признаки от самого ряда и даты

Временной ряд имеет выраженные колебания, локальные пики и просадки, а также заметное изменение уровня значений во времени.  
В данных присутствует структура, похожая на сезонность, поэтому использование только случайного разбиения было бы некорректным.  
Для такой задачи важно учитывать порядок наблюдений и не допускать утечки информации из будущего.  
Перед обучением в ноутбуке выполнен sanity-check: размер датасета, диапазон дат, проверка пропусков и базовый график ряда.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

Разбиение `train / validation / test` выполнено строго по времени.

Использовалась следующая схема:

- `train` — первые `70%` наблюдений
- `validation` — следующие `15%`
- `test` — последние `15%`

Границы split выводятся в ноутбуке явно, а также визуализируются на графике `series_split.png`.

`Random split` для временного ряда некорректен, потому что при случайном перемешивании модель может обучаться на наблюдениях, которые хронологически относятся к будущему относительно `validation` или `test`. Это приводит к `data leakage` и искусственно завышает качество модели.

### 4.2. Признаки для baseline-моделей

Для baseline-моделей использовались следующие лаговые признаки:

- `lag_1`
- `lag_7`
- `lag_14`

Использовались следующие rolling-признаки:

- `rolling_mean_7`
- `rolling_std_7`

Использовались следующие календарные признаки:

- `day_of_week`

Пропуски, возникающие после построения лаговых и rolling-признаков, удалялись до обучения табличной модели.

Масштабирование использовалось там, где это было необходимо.  
`StandardScaler` обучался только на `train`, после чего применялся к `validation` и `test` через `transform`.  
Это исключает утечку информации из будущих частей временного ряда.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- `B1 (naive-last)`: прогноз равен последнему известному значению ряда.
- `B2 (moving-average)`: прогноз равен среднему значению по предыдущему окну наблюдений.
- `B3 (ridge-lag-features)`: линейная модель `Ridge`, использующая лаговые, rolling- и календарные признаки.
- `R1 (gru-forecast)`: рекуррентная нейросеть `GRU`, работающая на оконном представлении временного ряда.

Дополнительно:

- Основная метрика выбора лучшей модели: `MAE` на `validation`
- Дополнительные метрики: `RMSE`, `MAPE`
- `window_size` для `GRU`: `14`
- Batch size: `64`
- Максимальное число эпох: `30`
- Optimizer: `Adam`
- Learning rate: `0.001`

Лучшая модель сохранялась по минимальному значению `validation MAE`.  
Для `GRU` сохранялся лучший `state_dict()` в файл `./artifacts/best_gru.pt`, а параметры модели — в `./artifacts/best_gru_config.json`.

## 7. Анализ

Наивный baseline `B1` оказался surprisingly сильным. Это характерно для временных рядов, в которых соседние значения сильно связаны между собой, а предыдущее значение уже несёт большую часть полезной информации для краткосрочного прогноза.

Baseline `B2` на основе скользящего среднего показал худший результат. Причина в том, что усреднение сглаживает ряд и ухудшает способность модели реагировать на резкие изменения, локальные пики и просадки.

Использование лаговых, rolling- и календарных признаков в `B3` позволило модели `Ridge` учитывать структуру ряда лучше, чем `moving-average`. Такая модель умеет использовать больше информации, чем просто последнее или среднее значение, поэтому её качество оказалось заметно выше `B2`.

При этом `Ridge` не превзошла лучший baseline `B1`. Это означает, что для данного ряда краткосрочная зависимость от последнего значения выражена очень сильно, а линейная комбинация признаков не даёт достаточного выигрыша.

`GRU` показала лучший результат среди всех подходов. Это объясняется тем, что рекуррентная модель лучше работает с последовательной структурой данных и способна учитывать взаимосвязь нескольких последних наблюдений в окне.

На графике прогноза видно, что `GRU` в целом хорошо повторяет форму ряда и отслеживает основные колебания. При этом самые сложные ошибки возникают в точках с резкими скачками или быстрыми локальными изменениями.

В задаче такого типа утечки данных могли бы появиться при неправильном построении лагов, при использовании rolling-признаков без сдвига, при обучении scaler на всех данных сразу или при выборе модели по `test`. Поэтому все преобразования выполнялись с учётом временного порядка.

Выбор лучшей модели по `validation`, а не по `test`, принципиален, потому что `test` должен оставаться полностью отложенной выборкой для финальной проверки. Если использовать `test` на этапе выбора модели, оценка перестаёт быть честной.

Итоговая картина показывает, что даже простые baseline-модели для временных рядов являются очень важной точкой отсчёта. Без их сравнения можно переоценить реальную пользу более сложной нейросетевой модели.

## 8. Итоговый вывод

Для данной задачи в качестве основного подхода я бы выбрал `GRU`, так как она показала лучшее качество на `validation` и сохранила его на `test`.

Если ряд относительно простой и краткосрочная зависимость от предыдущего значения очень сильная, то в качестве базового решения может быть достаточно наивного baseline или простой модели на лаговых признаках.

Переходить к `GRU` и другим нейросетям имеет смысл тогда, когда нужно лучше учитывать последовательную структуру ряда и более сложные зависимости между наблюдениями.

Главный вывод этой работы состоит в том, что для временных рядов критически важна корректная валидация по времени. `Temporal split` обязателен, а использование `random split` в такой постановке приводит к некорректной оценке качества модели.
